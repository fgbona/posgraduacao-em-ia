# Deployment com alta disponibilidade e segurança
apiVersion: apps/v1
kind: Deployment
metadata:
  name: chronos-api
  namespace: production
  labels:
    app: chronos-api
    version: v1
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: chronos-api
  template:
    metadata:
      labels:
        app: chronos-api
        version: v1
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/metrics"
    spec:
      # Segurança a nível de Pod
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        runAsGroup: 3000
        fsGroup: 2000
        seccompProfile:
          type: RuntimeDefault

      # Anti-afinidade para distribuição entre nós
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - chronos-api
              topologyKey: kubernetes.io/hostname

      # Prioridade da aplicação
      priorityClassName: high-priority

      # Tolerâncias para node taints
      tolerations:
      - key: "production"
        operator: "Equal"
        value: "true"
        effect: "NoSchedule"

      # Service Account com RBAC mínimo
      serviceAccountName: chronos-api
      automountServiceAccountToken: true

      containers:
      - name: api
        # Imagem versionada (nunca usar latest)
        image: chronos-api:1.2.3
        imagePullPolicy: IfNotPresent

        ports:
        - name: http
          containerPort: 8080
          protocol: TCP

        # Variáveis de ambiente
        env:
        # Configurações não sensíveis via ConfigMap
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: chronos-api-config
              key: LOG_LEVEL
        - name: ENVIRONMENT
          valueFrom:
            configMapKeyRef:
              name: chronos-api-config
              key: ENVIRONMENT
        - name: API_PORT
          valueFrom:
            configMapKeyRef:
              name: chronos-api-config
              key: API_PORT

        # Secrets via referência (melhor que plaintext)
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: chronos-api-secrets
              key: db-password
        - name: JWT_SECRET
          valueFrom:
            secretKeyRef:
              name: chronos-api-secrets
              key: jwt-secret

        # Resource Requests (garantem recursos mínimos)
        resources:
          requests:
            cpu: "100m"
            memory: "256Mi"
          # Resource Limits (evitam consumo excessivo)
          limits:
            cpu: "500m"
            memory: "512Mi"

        # Liveness Probe - reinicia o container se aplicação travar
        livenessProbe:
          httpGet:
            path: /health/live
            port: http
          initialDelaySeconds: 30
          periodSeconds: 10
          timeoutSeconds: 5
          failureThreshold: 3

        # Readiness Probe - remove do loadbalancer se não pronta
        readinessProbe:
          httpGet:
            path: /health/ready
            port: http
          initialDelaySeconds: 10
          periodSeconds: 5
          timeoutSeconds: 3
          failureThreshold: 2

        # Startup Probe - aguarda aplicação iniciar (apps lentas)
        startupProbe:
          httpGet:
            path: /health/startup
            port: http
          initialDelaySeconds: 0
          periodSeconds: 2
          timeoutSeconds: 3
          failureThreshold: 30

        # Segurança a nível de Container (CIS Benchmark)
        securityContext:
          allowPrivilegeEscalation: false
          capabilities:
            drop:
            - ALL
          readOnlyRootFilesystem: true
          runAsNonRoot: true
          runAsUser: 1000

        # Volume mounts para diretórios de escrita
        volumeMounts:
        - name: tmp
          mountPath: /tmp
        - name: cache
          mountPath: /app/cache
        - name: logs
          mountPath: /app/logs

        # Probes de segurança
        livenessProbe:
          exec:
            command:
            - /bin/sh
            - -c
            - "wget --quiet --tries=1 --spider http://localhost:8080/health/live || exit 1"
          initialDelaySeconds: 30
          periodSeconds: 10
          timeoutSeconds: 5
          failureThreshold: 3

      # Volumes temporários e efêmeros
      volumes:
      - name: tmp
        emptyDir:
          medium: Memory
          sizeLimit: 100Mi
      - name: cache
        emptyDir:
          sizeLimit: 200Mi
      - name: logs
        emptyDir:
          sizeLimit: 500Mi

      # Tempo de grace para shutdown elegante
      terminationGracePeriodSeconds: 30

      # DNS Policy para maior segurança
      dnsPolicy: ClusterFirst

# Modernização do Deployment Chronos API

## Resumo Executivo

O deployment foi completamente refatorado para atender às padronizações atuais da empresa e às melhores práticas de segurança do CIS Kubernetes Benchmark. A nova arquitetura garante alta disponibilidade, segurança robusta, e gerenciamento adequado de recursos.

---

## Mudanças Principais

### 1. **Alta Disponibilidade**

#### Antes:
```yaml
replicas: 1  # Único pod - ponto único de falha
```

#### Depois:
```yaml
replicas: 3  # Distribuição entre múltiplos nós
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0  # Zero downtime durante updates
```

**Benefícios:**
- Tolerância a falhas de nós
- Rolling updates sem interrupção de serviço
- PodDisruptionBudget garante mínimo 2 pods disponíveis durante manutenção

---

### 2. **Versionamento de Imagem**

#### Antes:
```yaml
image: chronos-api:latest  # ❌ RISCO: tag sempre muda
```

#### Depois:
```yaml
image: chronos-api:1.2.3   # ✅ Versão específica
imagePullPolicy: IfNotPresent
```

**Por quê:**
- `latest` é impredizível em produção
- Facilita rollbacks
- Atende CIS 5.2.2 e 5.2.3

---

### 3. **Secrets Fora do Manifest**

#### Antes:
```yaml
env:
- name: DB_PASSWORD
  value: "P@ssw0rd2023!"  # ❌ Plaintext no Git!
- name: JWT_SECRET
  value: "hvt-jwt-prod-secret"
```

#### Depois:

**Secret separado:**
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: chronos-api-secrets
stringData:
  db-password: "P@ssw0rd2023!"
  jwt-secret: "hvt-jwt-prod-secret"
```

**Referência no Deployment:**
```yaml
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: chronos-api-secrets
      key: db-password
```

**Implementação:**
```bash
# Criar secret (em produção, use um secrets manager):
kubectl create secret generic chronos-api-secrets \
  --from-literal=db-password=<valor-real> \
  --from-literal=jwt-secret=<valor-real> \
  -n production
```

---

### 4. **Resource Requests & Limits**

#### Antes:
```yaml
# Sem especificação - node poderia ficar sem recursos
```

#### Depois:
```yaml
resources:
  requests:
    cpu: "100m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

**O que cada um faz:**

| Campo | Propósito | Impacto |
|-------|----------|--------|
| **requests.cpu** | Kubernetes reserva este CPU para o pod | Garante performance mínima |
| **requests.memory** | Kubernetes reserva esta RAM | Pod é eviccionado se exceder |
| **limits.cpu** | CPU máxima que o pod pode usar | Throttling se exceder |
| **limits.memory** | RAM máxima que o pod pode usar | Pod é morto se exceder (OOMKill) |

**Valores recomendados (ajuste conforme teste de carga):**
```
Aplicação leve:
  requests: cpu=50m, memory=128Mi
  limits: cpu=250m, memory=256Mi

Aplicação média (como Chronos):
  requests: cpu=100m, memory=256Mi
  limits: cpu=500m, memory=512Mi

Aplicação pesada:
  requests: cpu=500m, memory=1Gi
  limits: cpu=2000m, memory=2Gi
```

---

### 5. **Health Probes**

#### Antes:
```yaml
# Sem probes - Kubernetes não sabe se app está saudável
```

#### Depois:

**Startup Probe** (aguarda app iniciar):
```yaml
startupProbe:
  httpGet:
    path: /health/startup
    port: http
  failureThreshold: 30
  periodSeconds: 2
  # Permite até 60s para app iniciar (30 * 2s)
```

**Readiness Probe** (pronta para receber tráfego):
```yaml
readinessProbe:
  httpGet:
    path: /health/ready
    port: http
  initialDelaySeconds: 10
  periodSeconds: 5
  failureThreshold: 2
  # Se falhar 2x, remove do service (mas pod continua rodando)
```

**Liveness Probe** (está viva):
```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: http
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
  # Se falhar 3x, mata o pod e restarted
```

**Sequência típica:**
```
Pod inicia
    ↓
Startup probe (até 60s)
    ↓
Pod pronto
    ↓
Readiness probe + Liveness probe (continuamente)
```

**Implementação na aplicação (pseudo-código):**
```javascript
// Exemplo Node.js/Express
app.get('/health/startup', (req, res) => {
  // Verifica se app iniciou (migrations, conexões críticas)
  if (app.isInitialized) res.status(200).json({status: 'starting'});
  else res.status(503).json({status: 'not ready'});
});

app.get('/health/ready', (req, res) => {
  // Verifica se pronta para tráfego (DB, cache, deps)
  if (db.isConnected && cache.isReady) res.status(200).json({status: 'ready'});
  else res.status(503).json({status: 'not ready'});
});

app.get('/health/live', (req, res) => {
  // Verifica se ainda está viva (heartbeat simples)
  res.status(200).json({status: 'alive', timestamp: Date.now()});
});
```

### 6. **SecurityContext Não-Root**

#### Antes:
```yaml
# Sem especificação - roda como root (UID 0)  ❌ CIS Violation
```

#### Depois:

**Pod-level security:**
```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 3000
  fsGroup: 2000
  seccompProfile:
    type: RuntimeDefault
```

**Container-level security:**
```yaml
securityContext:
  allowPrivilegeEscalation: false
  capabilities:
    drop:
    - ALL
  readOnlyRootFilesystem: true
  runAsNonRoot: true
  runAsUser: 1000
```

**O que cada um faz:**

| Config | Impacto | CIS Benchmark |
|--------|--------|---|
| `runAsNonRoot: true` | Obriga UID > 0 | 5.2.1 |
| `runAsUser: 1000` | UID específico (não usar 0) | 5.2.2 |
| `drop: ALL` | Remove TODAS as Linux capabilities | 5.2.7, 5.2.8, 5.2.9 |
| `allowPrivilegeEscalation: false` | Não permite escalação de privilege | 5.2.5 |
| `readOnlyRootFilesystem: true` | Filesystem raiz como read-only | 5.2.6 |

**Como rodar containers assim:**

1. Criar usuário não-root na imagem:
```dockerfile
FROM node:18-alpine
RUN addgroup -g 1000 appuser && \
    adduser -D -u 1000 -G appuser appuser
USER appuser
COPY --chown=appuser:appuser . /app
WORKDIR /app
```

2. Preparar diretórios de escrita:
```yaml
volumeMounts:
- name: tmp
  mountPath: /tmp
- name: cache
  mountPath: /app/cache
- name: logs
  mountPath: /app/logs

volumes:
- name: tmp
  emptyDir:
    medium: Memory
    sizeLimit: 100Mi
```

## Monitoramento Recomendado

Adicione alertas para:
- **Pod restarts**: `rate(kube_pod_container_status_restarts_total[1h]) > 0`
- **Memory pressure**: Pod usando >80% do limit
- **CPU throttling**: Container sendo throttled frequentemente
- **Failed health probes**: Readiness falhas contínuas
- **Pending pods**: Pods esperando por recursos

#!/bin/bash

###############################################################################
# Script de Deployment Seguro - Chronos API
#
# Uso: ./deploy-chronos-api.sh [stage]
# Stages: validate, secrets, deploy, verify, rollback
###############################################################################

set -euo pipefail

# Cores para output
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
NC='\033[0m' # No Color

# Variáveis
NAMESPACE="production"
DEPLOYMENT_NAME="chronos-api"
MANIFEST_FILE="chronos-api-deployment.yaml"
KUBECONFIG="${KUBECONFIG:-$HOME/.kube/config}"
TIMEOUT=300
LOG_PREFIX="[$(date +'%Y-%m-%d %H:%M:%S')]"

###############################################################################
# Funções utilitárias
###############################################################################

log_info() {
    echo -e "${BLUE}${LOG_PREFIX} ℹ️  INFO${NC}: $1"
}

log_success() {
    echo -e "${GREEN}${LOG_PREFIX} ✅ SUCCESS${NC}: $1"
}

log_warning() {
    echo -e "${YELLOW}${LOG_PREFIX} ⚠️  WARNING${NC}: $1"
}

log_error() {
    echo -e "${RED}${LOG_PREFIX} ❌ ERROR${NC}: $1"
}

###############################################################################
# Validação de pré-requisitos
###############################################################################

validate_prerequisites() {
    log_info "Validando pré-requisitos..."

    # Verificar kubectl
    if ! command -v kubectl &> /dev/null; then
        log_error "kubectl não encontrado. Instale kubectl antes de continuar."
        exit 1
    fi
    log_success "kubectl encontrado: $(kubectl version --client --short)"

    # Verificar kubeconfig
    if [ ! -f "$KUBECONFIG" ]; then
        log_error "Kubeconfig não encontrado em $KUBECONFIG"
        exit 1
    fi
    log_success "Kubeconfig encontrado"

    # Verificar conexão com cluster
    if ! kubectl cluster-info &> /dev/null; then
        log_error "Não foi possível conectar ao cluster Kubernetes"
        exit 1
    fi
    log_success "Conectado ao cluster: $(kubectl config current-context)"

    # Verificar namespace
    if ! kubectl get namespace "$NAMESPACE" &> /dev/null; then
        log_warning "Namespace '$NAMESPACE' não existe. Criando..."
        kubectl create namespace "$NAMESPACE"
        log_success "Namespace '$NAMESPACE' criado"
    fi

    # Verificar manifest
    if [ ! -f "$MANIFEST_FILE" ]; then
        log_error "Manifest '$MANIFEST_FILE' não encontrado"
        exit 1
    fi
    log_success "Manifest encontrado: $MANIFEST_FILE"
}

###############################################################################
# Validação do Manifest
###############################################################################

validate_manifest() {
    log_info "Validando manifest YAML..."

    # Validar sintaxe YAML
    if ! kubectl apply -f "$MANIFEST_FILE" --dry-run=client &> /dev/null; then
        log_error "Manifest YAML inválido"
        kubectl apply -f "$MANIFEST_FILE" --dry-run=client
        exit 1
    fi
    log_success "Sintaxe YAML válida"

    # Validar no servidor
    if ! kubectl apply -f "$MANIFEST_FILE" --dry-run=server -n "$NAMESPACE" &> /dev/null; then
        log_error "Manifest inválido para o servidor"
        kubectl apply -f "$MANIFEST_FILE" --dry-run=server -n "$NAMESPACE"
        exit 1
    fi
    log_success "Validação do servidor passou"

    # Extrair imagem e verificar versionamento
    IMAGE=$(grep -A 2 "image:" "$MANIFEST_FILE" | grep -oP 'chronos-api:\K[^"]+' | head -1)
    if [ -z "$IMAGE" ]; then
        log_error "Não foi possível extrair a imagem do manifest"
        exit 1
    fi

    if [ "$IMAGE" = "latest" ]; then
        log_error "Imagem com tag 'latest' detectada. Use tags de versão específicas."
        exit 1
    fi
    log_success "Imagem versionada corretamente: chronos-api:$IMAGE"

    # Verificar security context
    if ! grep -q "runAsNonRoot: true" "$MANIFEST_FILE"; then
        log_error "SecurityContext 'runAsNonRoot: true' não encontrado"
        exit 1
    fi
    log_success "SecurityContext não-root configurado"

    # Verificar resource limits
    if ! grep -q "limits:" "$MANIFEST_FILE"; then
        log_error "Resource limits não encontrados"
        exit 1
    fi
    log_success "Resource limits configurados"

    # Verificar health probes
    for probe in "livenessProbe" "readinessProbe"; do
        if ! grep -q "$probe:" "$MANIFEST_FILE"; then
            log_error "Health probe '$probe' não encontrada"
            exit 1
        fi
    done
    log_success "Health probes configurados"

    # Verificar secrets (não hardcoded)
    if grep -q 'value: "P@ssw0rd' "$MANIFEST_FILE"; then
        log_error "Credenciais detectadas em plaintext no manifest!"
        exit 1
    fi
    log_success "Nenhuma credencial em plaintext detectada"
}

###############################################################################
# Gerenciamento de Secrets
###############################################################################

setup_secrets() {
    log_info "Configurando secrets..."

    SECRET_NAME="chronos-api-secrets"

    # Verificar se secret já existe
    if kubectl get secret "$SECRET_NAME" -n "$NAMESPACE" &> /dev/null; then
        log_warning "Secret '$SECRET_NAME' já existe"
        read -p "Deseja atualizar o secret? (s/n) " -n 1 -r
        echo
        if [[ ! $REPLY =~ ^[Ss]$ ]]; then
            log_info "Pulando criação de secret"
            return 0
        fi
        kubectl delete secret "$SECRET_NAME" -n "$NAMESPACE"
    fi

    # Coletar valores
    read -sp "DB Password: " DB_PASSWORD
    echo
    read -sp "JWT Secret: " JWT_SECRET
    echo

    # Validar valores não vazios
    if [ -z "$DB_PASSWORD" ] || [ -z "$JWT_SECRET" ]; then
        log_error "Valores de secret não podem estar vazios"
        exit 1
    fi

    # Criar secret
    kubectl create secret generic "$SECRET_NAME" \
        --from-literal=db-password="$DB_PASSWORD" \
        --from-literal=jwt-secret="$JWT_SECRET" \
        -n "$NAMESPACE"

    log_success "Secret criado com sucesso"
}

###############################################################################
# Deployment
###############################################################################

deploy_manifest() {
    log_info "Iniciando deployment..."

    # Aplicar manifest
    if ! kubectl apply -f "$MANIFEST_FILE" -n "$NAMESPACE"; then
        log_error "Falha ao aplicar manifest"
        exit 1
    fi
    log_success "Manifest aplicado"

    log_info "Aguardando rollout..."

    # Aguardar deployment
    if kubectl rollout status deployment/"$DEPLOYMENT_NAME" \
        -n "$NAMESPACE" \
        --timeout="${TIMEOUT}s"; then
        log_success "Deployment completado com sucesso!"
        return 0
    else
        log_error "Deployment falhou ou timeout"
        return 1
    fi
}

###############################################################################
# Verificação Pós-Deployment
###############################################################################

verify_deployment() {
    log_info "Verificando deployment..."

    # Informações do deployment
    log_info "Status do deployment:"
    kubectl get deployment "$DEPLOYMENT_NAME" -n "$NAMESPACE"

    log_info "Pods em execução:"
    kubectl get pods -n "$NAMESPACE" -l app="$DEPLOYMENT_NAME"

    log_info "Detalhes da replicas:"
    DESIRED=$(kubectl get deployment "$DEPLOYMENT_NAME" -n "$NAMESPACE" \
        -o jsonpath='{.spec.replicas}')
    READY=$(kubectl get deployment "$DEPLOYMENT_NAME" -n "$NAMESPACE" \
        -o jsonpath='{.status.readyReplicas}')

    if [ "$DESIRED" -eq "$READY" ]; then
        log_success "Todas as replicas estão prontas ($READY/$DESIRED)"
    else
        log_error "Replicas não prontas ($READY/$DESIRED)"
        return 1
    fi

    # Verificar health probes
    log_info "Verificando health probes..."
    POD_NAME=$(kubectl get pods -n "$NAMESPACE" -l app="$DEPLOYMENT_NAME" \
        -o jsonpath='{.items[0].metadata.name}')

    if [ -z "$POD_NAME" ]; then
        log_error "Nenhum pod encontrado"
        return 1
    fi

    # Aguardar pod estar pronto
    kubectl wait --for=condition=Ready pod/"$POD_NAME" \
        -n "$NAMESPACE" \
        --timeout=60s

    log_info "Testando health endpoints..."

    # Port-forward para teste
    kubectl port-forward -n "$NAMESPACE" "pod/$POD_NAME" 8080:8080 &
    PF_PID=$!
    sleep 2

    # Testar endpoints
    for endpoint in "health/live" "health/ready"; do
        log_info "Testando /$endpoint..."
        if curl -f http://localhost:8080/$endpoint &> /dev/null; then
            log_success "✓ /$endpoint respondendo"
        else
            log_warning "✗ /$endpoint não respondendo (esperado se app não implementou)"
        fi
    done

    # Parar port-forward
    kill $PF_PID 2>/dev/null || true

    # Resource usage
    log_info "Resource usage:"
    kubectl top pods -n "$NAMESPACE" -l app="$DEPLOYMENT_NAME" || \
        log_warning "Metrics não disponíveis (aguarde métricas serem coletadas)"
}

###############################################################################
# Rollback
###############################################################################

rollback_deployment() {
    log_warning "Iniciando rollback..."

    if kubectl rollout undo deployment/"$DEPLOYMENT_NAME" -n "$NAMESPACE"; then
        log_success "Rollback iniciado"

        kubectl rollout status deployment/"$DEPLOYMENT_NAME" \
            -n "$NAMESPACE" \
            --timeout="${TIMEOUT}s"

        log_success "Rollback completado"
    else
        log_error "Falha ao fazer rollback"
        exit 1
    fi
}

###############################################################################
# Menu principal
###############################################################################

show_help() {
    cat << EOF
Uso: $0 [comando]

Comandos:
  validate     - Valida pré-requisitos e manifest
  secrets      - Cria/atualiza secrets
  deploy       - Executa deployment completo
  verify       - Verifica status do deployment
  rollback     - Faz rollback para versão anterior
  full         - Valida, configura secrets, deploya e verifica (recomendado)

Exemplos:
  $0 validate
  $0 full
  $0 rollback
EOF
    exit 0
}

###############################################################################
# Main
###############################################################################

main() {
    COMMAND="${1:-full}"

    case "$COMMAND" in
        validate)
            validate_prerequisites
            validate_manifest
            ;;
        secrets)
            validate_prerequisites
            setup_secrets
            ;;
        deploy)
            validate_prerequisites
            validate_manifest
            deploy_manifest
            ;;
        verify)
            validate_prerequisites
            verify_deployment
            ;;
        rollback)
            validate_prerequisites
            rollback_deployment
            ;;
        full)
            validate_prerequisites
            validate_manifest
            setup_secrets
            deploy_manifest
            verify_deployment
            log_success "✨ Deployment completo e verificado com sucesso!"
            ;;
        help|-h|--help)
            show_help
            ;;
        *)
            log_error "Comando desconhecido: $COMMAND"
            show_help
            ;;
    esac
}

# Executar
main "$@"

## Troubleshooting Comum

### Problema: Pods em Pending

```bash
# Causa provável: Recursos insuficientes ou limites de namespace

# Verificar:
kubectl describe pod <POD_NAME> -n production

# Procurar por "Events" > "FailedScheduling"

# Solução:
# 1. Aumentar capacidade do cluster
# 2. Reduzir requests/limits
# 3. Ajustar resource quotas do namespace
```

### Problema: Pods em CrashLoopBackOff

```bash
# Causa: Aplicação está falhando imediatamente

# Verificar logs:
kubectl logs <POD_NAME> -n production
kubectl logs <POD_NAME> -n production --previous  # Log anterior (antes do crash)

# Soluções comuns:
# 1. Verificar health probes (startupProbe muito rigoroso?)
# 2. Verificar configuração de secrets/env vars
# 3. Verificar permissões de filesystem
# 4. Aumentar initialDelaySeconds nas probes
```

### Problema: Readiness Probe Falhando

```bash
# Causa: Aplicação não consegue atingir estado "ready"

# Verificar manualmente:
kubectl exec -it <POD_NAME> -n production -- curl localhost:8080/health/ready

# Ver eventos do pod:
kubectl describe pod <POD_NAME> -n production | grep -A 5 "Readiness"

# Soluções:
# 1. Aumentar initialDelaySeconds
# 2. Verificar dependências (DB, cache, etc)
# 3. Revisar lógica de health check na aplicação
```

### Problema: Liveness Probe Matando o Pod

```bash
# Causa: Container é reiniciado frequentemente

# Monitorar:
kubectl get pods -n production -l app=chronos-api
# Ver coluna RESTARTS incrementando

# Verificar:
kubectl logs <POD_NAME> -n production --previous

# Aumentar failureThreshold ou period:
# livenessProbe:
#   failureThreshold: 5  # (aumentado de 3)
#   periodSeconds: 15    # (aumentado de 10)
```

### Problema: Memory Limit Exceeded

```bash
# Sintoma: Pods frequentemente mortos (OOMKilled)

# Verificar:
kubectl describe pod <POD_NAME> -n production | grep -i "oom\|memory"

# Ver uso real:
kubectl top pod <POD_NAME> -n production --containers

# Soluções:
# 1. Aumentar limits.memory no deployment
# 2. Otimizar aplicação
# 3. Adicionar HPA para escalar em vez de usar memória extra
```

## Rollback

### Automático (health probe falha)

```bash
# Se readiness probe falhar, Kubernetes automaticamente:
# 1. Para de enviar tráfego para o pod
# 2. Se persistir, pode deletar o pod

kubectl rollout history deployment/chronos-api -n production

# Saída:
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         <none>
```

### Manual (problema detectado)

```bash
# Ver histórico
kubectl rollout history deployment/chronos-api -n production

# Voltar para versão anterior
kubectl rollout undo deployment/chronos-api -n production

# Voltar para revisão específica
kubectl rollout undo deployment/chronos-api -n production --to-revision=1

# Verificar status
kubectl rollout status deployment/chronos-api -n production

# Se tudo OK
kubectl rollout pause deployment/chronos-api -n production  # Pausar updates
kubectl rollout resume deployment/chronos-api -n production # Retomar
```

## Riscos e Mitigações

| Risco | Probabilidade | Impacto | Mitigação |
|-------|---|---|---|
| Health probes muito rigorosas causam restarts | Média | Médio | Aumentar initialDelaySeconds/failureThreshold |
| Memory leak não detectado | Baixa | Médio | Monitorar por 1 semana; HPA como fallback |
| Secrets não migradas corretamente | Baixa | Alto | Testar em staging; validar acesso antes de deploy |
| NetworkPolicy bloqueia tráfego legítimo | Média | Alto | Testar isolamento em staging |
| Rollback lento/problemático | Baixa | Médio | Testar rollback em staging |

## Suporte e Escalação

### Problemas Comuns

**Pods em Pending:**
```bash
kubectl describe pod <POD_NAME> -n production
# Ver FailedScheduling nos events
# Causa: Recursos insuficientes
# Solução: Aumentar capacity ou reduzir requests
```

**Pods em CrashLoopBackOff:**
```bash
kubectl logs <POD_NAME> -n production
kubectl logs <POD_NAME> -n production --previous
# Causa: Aplicação falhando ao iniciar
# Solução: Revisar logs, aumentar probes
```

**Readiness Probe Falhando:**
```bash
kubectl exec -it <POD_NAME> -n production -- curl localhost:8080/health/ready
# Causa: Dependências (DB, cache) não prontas
# Solução: Aumentar initialDelaySeconds ou revisar saúde das dependências
```
